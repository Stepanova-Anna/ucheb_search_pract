## Этап 1. Выбор темы и постановка задачи

### Тема исследования
**Алгоритм маршрутизации в сети с учётом динамической нагрузки**

**Проектное решение**
Разработать и программно реализовать **адаптивный протокол маршрутизации**, который динамически перестраивает пути передачи данных в зависимости от текущей загрузки узлов и каналов связи, минимизируя потери пакетов и задержки.

**Решаемая проблема**
- Традиционные алгоритмы (OSPF, RIP) используют статические метрики (например, число хопов), что приводит к перегрузке одних каналов при простое других.
- В реальных сетях (IoT, корпоративные, mesh-сети) трафик неравномерен, а отказы узлов происходят часто.
- Отсутствие быстрой адаптации → рост времени отклика, джиттер и потеря пакетов.

**Ключевые особенности реализации**
1. **Гибридная метрика** – комбинация из:
   - загрузки канала (% от пропускной способности),
   - времени ожидания в очереди (RTT),
   - коэффициента потерь пакетов.
2. **Локальное обновление таблиц маршрутизации** – без полной пересылки всей топологии (упрощённый вариант Link State + агенты соседства).
3. **Механизм обратной связи** – каждый узел периодически отправляет соседям метрики качества соединения.
4. **Простая конфигурация** – возможность работы без центрального контроллера (децентрализованный режим).

**Ожидаемые преимущества перед аналогами**
| Аналог | Недостаток | Наше решение |
|--------|-------------|----------------|
| OSPF   | Статические веса, медленная сходимость | Динамические веса + локальная коррекция за секунды |
| RIP    | Ограничение в 15 хопов, не учитывает загрузку | Отсутствие ограничений на длину пути, учёт нагрузки |
| BGP    | Сложная настройка, не оптимизирован под быструю смену трафика | Децентрализован, автоматическая подстройка |

**Потенциальный эффект**:
- Снижение средней задержки на **20–40%** в условиях переменного трафика.
- Уменьшение числа перегруженных линий на **30–50%**.
- Повышение отказоустойчивости – перестроение маршрута за <1 секунды.

**Области применения**
- Mesh-сети (умный дом, уличное видеонаблюдение).
- Корпоративные Wi-Fi-сети с высокой плотностью пользователей.
- Резервные каналы ЦОД (адаптивный балансировщик трафика).

## Этап 2. Определение критериев оценки

**Цель оценки**
Подтвердить, что разработанный адаптивный алгоритм маршрутизации работает корректно и даёт измеримые преимущества над статическими аналогами (OSPF, RIP) в условиях переменной нагрузки.

**Критерии оценки**

### 1. Корректность (функциональная полнота)

| Критерий | Способ проверки | Пороговое значение |
|----------|----------------|---------------------|
| Доставка пакета при отсутствии отказов | Моделирование сети (Mininet / ns-3) | 100% пакетов доставлено |
| Перестроение маршрута при отказе узла | Эмуляция выхода узла из строя | Время восстановления < 1 с |
| Отсутствие петель маршрутизации | Логирование трассировки путей | 0 петель в любой момент времени |
| Стабильность при флуктуациях нагрузки | Циклическое изменение трафика (sin-модель) | Нет циклической «раскачки» маршрутов |
| Обработка пограничных случаев | Пустая сеть (1 узел), переполнение буферов | Без падений / бесконечных циклов |

### 2. Эффективность (производительность и ресурсы)

| Критерий | Единица измерения | Методика измерения | Целевой показатель |
|----------|------------------|--------------------|--------------------|
| Время сходимости после изменения топологии | мс | Время от события (отказ/добавление узла) до стабилизации таблиц | < 500 мс |
| Средняя задержка доставки пакета | мс | Сквозной RTT через всю сеть | На 20–40% ниже, чем у OSPF |
| Джиттер (вариация задержки) | мс | Стандартное отклонение RTT | На 25% ниже, чем у RIP |
| Потери пакетов при перегрузке | % | Отправка трафика > пропускной способности канала | < 1% (у OSPF > 5%) |
| Потребление памяти на узел | МБ | Размер таблиц маршрутизации + метрик соседей | < 5 МБ для сети из 100 узлов |
| Процессорное время на пересчёт маршрутов | мс / событие | Замер CPU time на узле при изменении топологии | < 10 мс на одно обновление |
| Накладной тракт служебных сообщений | Кбит/с | Объём обновлений метрик в единицу времени | < 5% от полезного трафика |

### 3. Масштабируемость

| Критерий | Диапазон проверки | Ожидаемое поведение |
|----------|-------------------|----------------------|
| Рост времени сходимости | 10, 50, 100, 200 узлов | Сублинейный рост (O(n log n) или лучше) |
| Рост потребления памяти | 10, 50, 100, 200 узлов | Линейный рост (O(n) по числу соседей) |
| Увеличение частоты обновлений | 0.1 Гц, 1 Гц, 10 Гц | Деградация не более 30% по задержкам |

### 4. Сравнительная эффективность (бенчмарк)

| Параметр | OSPF (статический) | RIP | Наш алгоритм (цель) |
|----------|--------------------|-----|---------------------|
| Время сходимости | 2–10 с | 30–60 с | < 0.5 с |
| Задержка при высокой нагрузке (50% канала) | 120 мс | 180 мс | < 90 мс |
| Потери пакетов при перегрузке | до 8% | до 12% | < 1% |
| Объём служебного трафика | Средний | Высокий | Низкий–Средний |

## Методика тестирования

1. **Среда**: эмулятор Mininet + Python (контроллер на Ryu или самостоятельная реализация).
2. **Топологии**:

# Этап 3. План эксперимента

## 1. Цель и задачи эксперимента

### Цель
Экспериментально подтвердить, что предложенный адаптивный алгоритм маршрутизации с динамической метрикой превосходит статические протоколы (OSPF, RIP) по задержке, потерям пакетов и времени сходимости в условиях переменной нагрузки и отказов узлов.

### Задачи
1. Реализовать прототип алгоритма в эмуляторе сети (Mininet).
2. Настроить эталонные реализации OSPF (FRRouting) и RIP для сравнительного тестирования.
3. Провести серию экспериментов с разными сценариями нагрузки и топологиями.
4. Собрать метрики: RTT, потери пакетов, время сходимости, потребление памяти/CPU.
5. Статистически сравнить результаты и проверить гипотезы.

---

## 2. Исходная гипотеза

### H0 (нулевая гипотеза)
Адаптивный алгоритм не даёт статистически значимого выигрыша по задержке и потерям пакетов по сравнению с OSPF/RIP при p < 0.05.

### H1 (альтернативная гипотеза)
Адаптивный алгоритм обеспечивает:
- **Снижение средней задержки** не менее чем на 20% при загрузке каналов > 50%.
- **Уменьшение потерь пакетов** в 3 и более раза в условиях перегрузки.
- **Время сходимости** менее 500 мс (против 2–10 с у OSPF).

### Дополнительные гипотезы
- **H2a**: Потребление памяти на узел не превышает 5 МБ для сети из 100 узлов.
- **H2b**: Накладной тракт служебных сообщений не превышает 5% от полезного.

---

## 3. Методы и методология

### 3.1 Тип эксперимента
**Сравнительный, контролируемый, изолированный** – эмуляция в виртуальной среде с повторяемыми условиями.

### 3.2 Методология
- Каждый эксперимент повторяется **10 раз** для статистической значимости.
- Топологии и сценарии нагрузки фиксируются (seed для генераторов случайных чисел).
- Одновременно запускаются три реализации (наш алгоритм, OSPF, RIP) на идентичных топологиях.
- Измерения автоматизированы (Python-скрипты сборщика метрик).

### 3.3 Метрики и инструменты

| Метрика | Инструмент | Частота сбора |
|---------|-----------|----------------|
| RTT (медиана, 95-й перцентиль) | `ping` (ICMP), `iperf3 -u` | Каждые 0.5 с |
| Потери пакетов | `iperf3` (UDP-поток) | За каждый прогон |
| Время сходимости | Логи событий + `tcpdump` | Для каждого изменения топологии |
| Потребление CPU (%) | `psutil` (Python) | Каждую 1 с |
| Потребление памяти (RSS, МБ) | `psutil` | Каждую 1 с |
| Служебный трафик (байт/с) | `tcpdump` + анализ заголовков | Непрерывно |

### 3.4 Программный стек
- **Эмулятор**: Mininet 2.3.0
- **Язык сценариев**: Python 3.10
- **Контроллер для нашего алгоритма**: Ryu 4.34 + самописный модуль
- **Эталонные протоколы**: FRRouting (OSPFv2), Quagga (RIP)
- **Генератор трафика**: iPerf3, `tcpreplay`
- **Анализ**: Pandas, SciPy (t-тест, ANOVA), Matplotlib/Seaborn

## 4. Условия проведения

### 4.1 Аппаратная среда
- Виртуальная машина (VirtualBox) на хосте с 8 vCPU, 16 ГБ RAM.
- Выделенные ресурсы: 4 vCPU, 8 ГБ RAM для Mininet.
- ОС: Ubuntu 22.04 LTS.

### 4.2 Сетевые параметры
- Виртуальные линки: пропускная способность от 10 до 100 Мбит/с.
- Задержка на линках: 2–20 мс (случайное распределение).
- Потери на линках (фон): 0% (кроме сценариев перегрузки).

### 4.3 Контролируемые переменные
- Тип топологии (фиксирован на время сравнения).
- Интенсивность трафика (Poisson-процесс с lambda = 10–100 пакетов/с).
- Длительность каждого прогона: 300 секунд.
- Время стабилизации перед замерами: 30 секунд.

## 5. Этапы проведения

### Этап 3.1 – Подготовка (2 дня)
- Реализация прототипа адаптивного алгоритма на Ryu.
- Написание скриптов автоматизации (запуск топологий, сбор метрик).
- Валидация корректности (ping all, проверка петель).

### Этап 3.2 – Пилотный эксперимент (1 день)
- Запуск на простой топологии (линейная из 5 узлов).
- Проверка, что все инструменты работают и метрики собираются.
- Корректировка параметров (частоты опроса, таймаутов).

### Этап 3.3 – Основная серия (3 дня)

| № | Сценарий | Топология | Переменная нагрузка | Что измеряется |
|---|----------|-----------|---------------------|----------------|
| 1 | Базовая корректность | Линейная (5 узлов) | Нет, фоновая 10% | Доставка 100%, отсутствие петель |
| 2 | Перегрузка одного канала | Звезда (1 центр + 8) | На центральный канал 120% от ёмкости | Потери, задержка, перестроение |
| 3 | Отказ узла | Случайный граф (20 узлов) | Узел отключается на 20-й секунде | Время сходимости, потери |
| 4 | Переменная нагрузка | Случайный граф (20 узлов) | Синусоидальная нагрузка (20–90%) | RTT, джиттер, служебный трафик |
| 5 | Масштабируемость | Линейная (10, 50, 100 узлов) | 50% на всех линках | Память, CPU, время сходимости |

Каждый сценарий повторяется **10 раз** для каждого протокола (наш, OSPF, RIP).  
Всего: 5 сценариев × 3 протокола × 10 повторов = **150 прогонов**.

### Этап 3.4 – Сбор и предобработка (1 день)
- Агрегация логов в CSV-файлы.
- Отбраковка выбросов (если аппаратный сбой).
- Нормализация временных рядов.

### Этап 3.5 – Анализ и статистика (1 день)
- Расчёт средних, медиан, 95% перцентилей.
- t-тест для сравнения средних (наш vs OSPF).
- Построение графиков: CDF задержек, ящики с усами.

## 6. Анализ результатов

### 6.1 Критерии принятия/отклонения H0

| Гипотеза | Условие принятия H1 |
|----------|----------------------|
| H1 (задержка) | Медиана RTT адаптивного алгоритма ≤ 0.8 × медианы RTT OSPF, p < 0.05 |
| H1 (потери) | Средние потери пакетов < 1% (у OSPF > 5%), p < 0.05 |
| H1 (сходимость) | Время сходимости < 500 мс в 95% случаев |
| H2a (память) | RSS на узел ≤ 5 МБ при 100 узлах |
| H2b (служебный трафик) | Доля служебных пакетов ≤ 5% от общего |

### 6.2 Планируемые визуализации

1. **CDF-график задержек** – на одном поле три кривые (наш, OSPF, RIP).
2. **Ящик с усами** для потерь пакетов по сценариям.
3. **Временной ряд** задержки при отказе узла (показать пик и восстановление).
4. **График «время сходимости vs размер сети»** (log-log).
5. **Таблица средних значений** с указанием p-value.

### 6.3 Возможные артефакты и их обработка

| Артефакт | Способ минимизации |
|----------|---------------------|
| Начальная «холодная» кэш-память | Прогрев 30 с перед замерами |
| Влияние фоновых процессов хоста | Запуск с `nice -n -20`, изоляция CPU через `taskset` |
| Синхронизация времени в эмуляторе | Использование виртуального времени Mininet (`--link tc`) |
| Случайные выбросы из-за переполнения очередей | Повтор 10 раз + отбрасывание выбросов по IQR |

### 6.4 Формат итогового отчёта

- **Таблица сравнения** всех метрик (среднее ± std).
- **Вывод**: принята или отклонена H0.
- **Ограничения эксперимента** (что не проверялось).
- **Рекомендации** по доработке алгоритма.
